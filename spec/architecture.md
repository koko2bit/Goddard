# Architecture

## Technology Stack

| Layer | Technology |
|-------|-----------|
| API / Webhooks / SSE | Cloudflare Workers |
| Real-time broadcast | Cloudflare Durable Objects |
| Database | Turso (SQLite at the Edge) + Drizzle ORM |
| Authentication | GitHub OAuth Device Flow |
| Package management | `pnpm` Workspaces |
| Distribution | `git-subrepo` ‚Üí standalone repositories |

## Component Responsibilities

### Backend (`backend/` + `github-app/`)

Hosted on Cloudflare Workers; acts as the control plane.

**Responsibilities:**
- Manage Device Flow authentication state and issue session tokens.
- Validate session tokens on every request.
- Receive and route GitHub webhook events (`pull_request`, `issue_comment`, `pull_request_review`).
- Apply automated reactions to events via Octokit (`goddard[bot]` identity).
- Broadcast routed events to subscribed CLI clients over Server-Sent Events (SSE).

**Real-time broadcast model:** Each repository is represented by a Cloudflare Durable Object. Webhook events are routed to the matching Durable Object, which broadcasts them over all open SSE connections for that repository. This provides per-repo fan-out with strong isolation.

**Schema authority:** A single schema module is the source of truth for table structure. Migrations are generated by diffing schema snapshots ‚Äî never written by hand.

---

### Core SDK (`sdk/`)

Package name: `@goddard-ai/sdk`. Zero runtime environment assumptions (browser, Node, Cloudflare).

**Design principle:** All platform capabilities are exposed through this package. CLI and agent loops are thin consumers ‚Äî they add UX and orchestration, not platform logic.

**Dependency injection:** Accepts a `TokenStorage` interface for flexible session token persistence (in-memory, file system, `localStorage`, etc.).

**Public API surface:**
- `sdk.pr.create(options)` ‚Äî sends PR-creation intent to the backend.
- `sdk.stream.subscribeToRepo(repo)` ‚Äî manages SSE lifecycle; normalizes frames into typed `EventEmitter` events (`comment`, `review`, `error`).

‚Üí See [`adr/001-sdk-first-architecture.md`](./adr/001-sdk-first-architecture.md) for the rationale behind this constraint.

---

### Interactive CLI (`cmd/`)

Package name: `@goddard-ai/cmd`. A thin consumer of `@goddard-ai/sdk`.

**Responsibilities:** Terminal UX, local `TokenStorage` implementation, and `.git/config` parsing for automatic `owner/repo` inference.

‚Üí See [`cli/interactive.md`](./cli/interactive.md) for the full command specification.

---

### Autonomous Loop (`cmd/` loop subcommands)

Managed through the `goddard loop` subcommand group.

**Responsibilities:** Config discovery, loop instantiation, `systemd` unit generation.

‚Üí See [`cli/loop.md`](./cli/loop.md), [`runtime-loop.md`](./runtime-loop.md), and [`configuration.md`](./configuration.md) for details.

## Deployment Model

**Target:** Cloudflare Workers (globally distributed, edge-native).

**Infrastructure prerequisites:**
- Turso database for persistence (users, sessions, installations).
- GitHub App registration with configured webhook URL pointing to the Worker.
- Cloudflare Durable Objects namespace for per-repo SSE fan-out.
- Production secrets managed via the Cloudflare secrets API.

**Local development:** An in-memory control plane is used for local testing without Turso or Cloudflare infrastructure. This is a development convenience ‚Äî not a supported production mode.

‚Üí See [`adr/002-edge-native-backend.md`](./adr/002-edge-native-backend.md) for the rationale behind the Cloudflare Workers choice.
